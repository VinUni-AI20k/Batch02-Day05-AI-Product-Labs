Q5 Lỗi giao dịch "chuyển tiền bị trừ 2 lần" ✕ Hard failure

✕ Path quan trọng nhất (user mất tiền) fail hoàn toàn — không có fallback, không có handoff CSKH

06

Path yếu nhất: Lỗi giao dịch

✕

Query 5 — "chuyển tiền bị trừ 2 lần" → System timeout

01

User context nguy hiểm — người dùng đang bị mất tiền thật, cần giải quyết ngay lập tức

02

Failure mode tệ nhất — system crash hoàn toàn, không có bất kỳ fallback nào

03

Tác động brand cao — trải nghiệm tệ nhất xảy ra đúng lúc user cần tin tưởng MoMo nhất

04

Fix đơn giản — không cần AI phức tạp, chỉ cần intent detection + hardcode redirect CSKH

Finding viết thành product decision

Khi user báo sự cố giao dịch có rủi ro mất tiền như "chuyển tiền bị trừ 2 lần", AI/product đi vào trạng thái tìm dữ liệu rồi timeout, chỉ yêu cầu user thử lại. Hậu quả là user không biết kênh khẩn cấp để recover tiền, niềm tin vào MoMo giảm đúng lúc cần hỗ trợ nhất. Lỗi thuộc layer intent routing + UX recovery + human handoff. Nên sửa bằng rule ưu tiên cao cho nhóm intent "mất tiền/lỗi giao dịch/hoàn tiền": bỏ qua search pipeline, trả lời bằng checklist an toàn, link CSKH, hotline và nút gọi/chat ngay.

07

Đề xuất to-be

As-is

  1. User báo lỗi chuyển tiền bị trừ 2 lần.
  2. Moni kích hoạt pipeline tìm dữ liệu.
  3. Hệ thống timeout và chỉ hiện nút Thử lại.
  4. User không có hotline, ticket, link CSKH hoặc hướng dẫn recovery.

To-be

  1. Detect intent rủi ro cao: mất tiền, trừ tiền, hoàn tiền, lỗi giao dịch.
  2. Bỏ qua search pipeline nếu confidence thấp hoặc hệ thống chậm.
  3. Trả lời bằng hướng dẫn an toàn, hotline, link CSKH và nút Gọi ngay/Chat ngay.
  4. Log correction và trạng thái handoff để support tiếp tục xử lý.

08

Kết luận

Moni có nền tốt — scope guard hoạt động, UI/UX nhắn tin mượt, loading state rõ ràng. Vấn đề cốt lõi không phải ở model mà ở data access và intent routing.

Promise "phân tích chi tiêu" chưa được fulfill vì thiếu bridge với transaction data. Path lỗi giao dịch cần ưu tiên fix ngay — đây là zero-tolerance failure. 3 low-confidence paths có thể cải thiện bằng clarification flow và scope tie-back, không cần thay model.

SPEC impact: SPEC cần thêm requirement bắt buộc cho high-risk transaction intent: detect trước khi search, fallback dưới 2 giây, handoff CSKH rõ ràng và log correction/retry để support xử lý tiếp.

Ưu tiên fix: Transaction error path → Clarification trước tư vấn → Data access bridge
